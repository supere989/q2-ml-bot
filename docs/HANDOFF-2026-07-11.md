## What this project is

ML-based Quake 2 bot: a PPO-trained LSTM policy (`models/policy.py`)
controls in-game bots over a UDP bridge (`harness/env.py` ‚Üî
`merge_mod/lithium`'s `ml_bridge.c`/`ml_obs.c` in `game.so`), with a
per-voxel-cell spatial reward system (`harness/spatial.py`, "the Voxel
Lattice") and a live, human-joinable deployment
(`tools/live_match_onnx.py`, ONNX Runtime, running in production on a VPS).
Full architecture diagram: `docs/architecture-map.svg`. Full narrative
detail on everything below: `README.md` ¬ß Known Issues / Roadmap.

## What was fixed this session (chronological, each independently verified)

1. **`ml_bridge.c :: ML_RecvAction()` bootstrap race** ‚Äî a non-blocking
   (`MSG_DONTWAIT`) check in the first-frame bootstrap path meant a solo
   live deployment could never escape it (needed the harness's UDP reply
   already buffered at the exact instant checked), so every action applied
   as a zero fallback forever. Training's 48-bot batched timing gave enough
   incidental slack to mostly avoid it, which is why this went unnoticed
   for so long. Fixed with a bounded 50ms real timeout. **Deployed to
   production**, reconciled into both machines' git history, pushed.
2. **`wsl-tunnel.service` (TensorBoard access)** ‚Äî silently broken on
   nobara this whole time (unrelated systemd `%d`/`LoadCredential`
   issue, see AGENTS.md). Fixed; same service added to procreator.
3. **Training data staleness** ‚Äî nobara's checkpoint/runs mirror was
   months stale. Now auto-synced every 15 min via
   `q2ml-training-sync.timer`. One-way, `--delete` OFF (nobara has unique
   recorded-human-play logs under `runs/` that must be preserved).
4. **`tools/behavior_clone_aim.py`** ‚Äî crashed on `KeyError: 'fire_logits'`;
   predates the model's fire decision becoming an autoregressive head
   (`fire_logits_for(feat, weapon_idx)`). Never successfully run before
   this fix. Fixed via teacher-forcing on the ground-truth weapon.

## The open problem: aim doesn't work, and here's the exact evidence trail

The deployed checkpoint (`checkpoints/CUR/policy_39929600.pt`, 39.9M steps)
moves and navigates fine but essentially never lands hits. Four independent
measurements confirmed this (not the bootstrap bug above ‚Äî that's fixed and
separately verified not to be the cause):

- Live yaw-tracking telemetry: mean facing error to a visible target 97.5¬∞,
  0/40 fire events within the reward's own 12¬∞ alignment threshold.
- Historical TensorBoard (`combat/kd_ratio` on run `ppo_CUR_1781839640`):
  plateaued ~1.5:1 for the entire second half of training, flat, no
  late-run improvement. `win_rate` ~0.1% throughout.
- Controlled `tools/evaluate_kd.py` re-test (isolated copy of the runtime
  dir, so the live trainer was never touched): 0.0 kd both before and after
  the bootstrap fix ‚Äî confirms this is a policy-quality problem, not the
  control-plane bug.
- **ML-vs-ML self-play** (`tools/ml_vs_ml.py`, written this session ‚Äî pits
  N ONNX policy instances against each other): the checkpoint fails
  identically against *itself* (1 kill/24 deaths vs. 5 kills/26 deaths over
  3000 steps), ruling out "the opponent is just good" as an explanation.

**Root cause**: pure PPO exploration never discovered the aim‚Üífire
association in 40M steps. The reward function has dozens of competing
terms (exploration, survival, spatial lattice, exchange dynamics, item
timing...); movement/survival reward is cheap and easy to accumulate,
aim is a narrow, hard-to-discover, sparse-reward skill, and gradient never
favored it.

**Fix in progress**: `tools/behavior_clone_aim.py`, a supervised warm-start
from a scripted geometric teacher (`yaw = atan2(rel_y, rel_x)` toward the
nearest visible enemy ‚Äî trivial trig, always correct). First run
(2026-07-11, `--synthetic` mode, default hyperparameters, ~20s of
training): mean facing error 97.5¬∞‚Üí81.2¬∞, within-12¬∞ rate 6.0%‚Üí8.0%,
**fires-while-aligned 0.0%‚Üí7.7%**. Directionally correct, not yet good.
New checkpoint: `checkpoints/policy_39929601.pt` /
`checkpoints/policy_latest.onnx` (WSL box + nobara mirror; not `CUR`, not
deployed).

## How to measure aim quality (reusable method)

Not (yet) landed as a proper tool ‚Äî was done via ad hoc instrumentation of
a scratch copy of `live_match_onnx.py` this session
(`tools/live_match_onnx_dbg.py` on the WSL box only, not committed ‚Äî
worth promoting to a real tool if this becomes a recurring check). The
method: log per-tick `(cur_yaw, nearest-visible-enemy bearing, commanded
yaw delta, fire flag)`, compute `yaw_err = ((bearing - cur_yaw + 180) % 360)
- 180`, then report mean/median `|yaw_err|`, % of frames within
`Q2_AIM_YAW_DEG` (12¬∞), and ‚Äî most diagnostic ‚Äî % of *fire* events that
happened within that threshold. Run with real opponents present
(`--n_bots 8`, NOT `--n_bots 2` ‚Äî see gotcha below) so there's something to
aim at. Cross-checked the yaw-convention math against the engine's actual
`AngleVectors()` (`q_shared.c`) before trusting it ‚Äî worth doing again if
you port this to a different observation layout.

## Gotchas (each cost real time to discover ‚Äî don't rediscover them)

- **`Q2_EXT_OBS=1` is required** for `CUR`/most current checkpoints (they
  were trained with it ‚Üí 219-dim obs, not the code default of 209).
  Omitting it is a silent `size mismatch for encoder.rest.0.weight` crash
  on load, not a training bug.
- **`ml2sk1` botlist (`n_bots=2`) doesn't spawn anything** on the WSL box ‚Äî
  no error, server just sits idle, harness times out waiting for obs. Root
  cause not investigated. Use `n_bots=4`+ sizing for anything needing
  multiple bot slots (including multi-ML-bot setups like `ml_vs_ml.py`).
- **`fire_logits` is not a plain key** in `Q2BotPolicy.forward()`'s output
  ‚Äî it's conditioned on the chosen weapon via
  `fire_logits_for(feat, weapon_idx)`. If you see `KeyError: 'fire_logits'`
  anywhere else, same fix pattern applies.
- **`q2ded` dlopens `lithium/game.so`**, not `gamex86_64.so` ‚Äî always copy
  the build output to that exact filename. Servers respawn per round, so a
  copied `game.so` goes live on the next round without touching a running
  trainer.
- **Never build/test C changes directly against `~/q2_lithium_merge`** while
  the trainer is running ‚Äî its `q2ded` servers share that exact path and
  will pick up whatever's there on their next round rotation. Use an
  isolated copy (`cp -r ~/q2_lithium_merge ~/q2_lithium_merge_eval`, point
  `Q2_ROOT` at the copy) for any experimentation, exactly like
  `evaluate_kd.py`/`ml_vs_ml.py` testing did this session.
- **`__DATE__`/`__TIME__` are embedded in `game.so`** (version strings in
  `g_save.c`/`lithium.c`) ‚Äî identical source produces different checksums
  on every rebuild. Don't use checksum equality to verify "did my change
  actually apply" ‚Äî diff the disassembly (`objdump -d`) or the source
  instead.
- **Python subprocess stdout is fully buffered when redirected to a file**
  ‚Äî always use `python3 -u` for anything backgrounded with `> log.txt`, or
  you'll see zero output until the process exits.
- **`pkill -f <pattern>` can self-match** if your own shell's command line
  (e.g. a later `grep` in the same compound command) contains the same
  pattern string ‚Äî kills the wrapper instead of the target, surfaces as a
  confusing SSH exit 255. Keep pkill patterns isolated to their own
  command, or use the `[x]pattern` bracket trick.
